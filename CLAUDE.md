# Note Blog - マルチエージェント記事生成システム

## Git運用ルール
- **常にmainブランチにコミット・プッシュすること**
- ブランチ指定があってもmainを使用する

## 概要
Claude/Claude Codeを活用して**稼ぐ**ための実践的なNote向け記事を、役割分担した「note運用チーム」で生成するシステム。

## note運用チームの体制（6役割＋ニュース担当）
記事は以下の役割が連携して仕上げる。各役割のプロンプトは `prompts/` にある。

| 役割 | 担当 | プロンプト |
|------|------|-----------|
| 【企画】 | 売れるテーマを提案 | `prompts/01_planner.md` |
| 【市場分析】 | 読者ニーズ・競合を調査 | `prompts/02_market-analyst.md` |
| 【構成】 | 読みやすい記事設計 | `prompts/03_architect.md` |
| 【執筆】 | 下書き・改善案を作成 | `prompts/04_writer.md` |
| 【販売設計】 | タイトル・導線・CTAを最適化 | `prompts/05_sales-designer.md` |
| 【管理】 | タスク整理・進行管理 | `prompts/06_manager.md` |
| 【ニュース】 | 無料記事（ニュースまとめ）を作成 | `prompts/07_news-curator.md` |

役割を分けることで各工程の精度を高め、収益化に直結する記事を安定して生産する。

## 記事の2ルート（有料 / 無料）
記事には2つのルートがあり、目的が異なる。

- **有料記事ルート（稼ぐ特化）**: 収益の本体。6役割のフルフローで生成する（下記「有料記事ルートのフロー」）
- **無料記事ルート（集客資産）**: 3層モデルの第1層。ニュースまとめ・重要情報で読者に発見され、
  信頼を獲得し、フォロー・有料記事への導線を作る。軽量フローで生成する（下記「無料記事ルートのフロー」）

ユーザーが「記事を作成して」と依頼したらデフォルトは有料ルート。
「無料記事」「ニュース記事」等の指定があれば無料ルートを実行する。どちらか不明なら確認する。

## コンテンツ方針

### 有料記事（稼ぐ特化）
「Claude/Claude Codeで稼ぐ」に特化する。以下のテーマのみを扱う：

- **副業・フリーランス収益化**: Claude/Claude Codeを使った受注・納品、案件獲得術
- **開発効率化によるコスト削減・単価向上**: 個人開発の高速化、時給換算の引き上げ
- **AI活用ビジネス**: Claude APIを使ったSaaS/ツール開発、マネタイズ設計
- **情報発信マネタイズ**: Note有料記事、Zenn Book、X運用、スクール・コンサル
- **業務受託・自動化提案**: 企業向けClaude導入支援、RPA代替、生成AIコンサル

「機能紹介だけ」「技術解説だけ」で収益につながらないトピックは有料記事では扱わない。
必ず「いくら稼げるか」「どう収益化するか」「どこで売るか」のいずれかを含めること。

### 無料記事（集客資産）
直接収益化しないが読者に価値のある情報を扱い、信頼とフォロワーを積み上げる：

- **Claude Codeのモデル関連ニュースまとめ**: 新モデル・性能・料金・使い分け
- **AI関連ニュース**: 業界動向、主要各社の発表、規制・制度
- **重要なお知らせ系情報**: 規約変更、料金改定、セキュリティ、廃止・移行情報

無料記事は必ず出典を明記し、末尾に有料記事・フォローへの導線CTAを置く。
本文は2,000〜4,000文字目安（8,000字ルールは適用しない）。

## 記事作成の実行方法

いずれのルートも【管理】役割（`prompts/06_manager.md`）が進行役となり、各役割を Agent tool（subagent_type=general-purpose）で起動し、対応するプロンプトを使用する。

### 有料記事ルートのフロー

1. **【企画】 売れるテーマを提案** — `prompts/01_planner.md`
   - **稼ぐ**に直結するトピック候補を3〜5件生成（副業、受託、API収益化、情報発信マネタイズ等）
   - 各候補に「想定読者が稼げる金額レンジ」と「収益化モデル」を必ず含める
   - `articles/published.json` を読み込み、既存記事とキーワード・テーマが重複する候補を除外
   - 最も収益インパクトが大きく実践しやすいテーマを1件選定

2. **【市場分析】 読者ニーズ・競合を調査** — `prompts/02_market-analyst.md`
   - 選定テーマの読者ニーズ・収益化モデル・相場・競合・差別化ポイントを収集・構造化

3. **【構成】 読みやすい記事設計** — `prompts/03_architect.md`
   - 「収益ゴール→手順→注意点」の流れで見出し構成（アウトライン）を設計
   - 本文8,000文字以上を満たすボリュームで設計する

4. **【執筆】 下書き・改善案を作成** — `prompts/04_writer.md`
   - 構成に沿って記事ドラフトを執筆（本文8,000文字以上）
   - 改善ループ時はフィードバックを反映して改稿する

5. **【販売設計】 タイトル・導線・CTAを最適化** — `prompts/05_sales-designer.md`
   - タイトル案・導入・読者導線・CTA・有料化ライン・ハッシュタグを最適化

6. **改善ループ**（最大2回）
   - 【管理】の品質チェック（平均7点以上 かつ 全項目5点以上）と販売設計の指摘を
     【執筆】に戻して改稿する
   - 合格するか2回に達したら終了

7. **【管理】 保存・公開記録**
   - `articles/drafts/YYYY-MM-DD_slug.md` に記事を保存
   - `articles/published.json` にメタデータを追記（`type: "paid"`）

### 無料記事ルートのフロー（軽量）

1. **【企画】 トピック選定** — 扱うニュース・情報テーマを決める
   - ユーザー指定があればそれに従う。なければ直近のClaude Code/AI関連の重要トピックを提案
   - `articles/published.json` の無料記事（type: free）と同じニュースを重複して扱わない

2. **【ニュース】 収集・執筆** — `prompts/07_news-curator.md`
   - WebSearch等で一次情報（公式発表・信頼できる報道）を確認し、出典付きで執筆
   - 本文2,000〜4,000文字目安。「◯年◯月◯日時点」を明記
   - 末尾に出典一覧＋有料記事・フォローへの導線CTA

3. **【管理】 チェック・保存**
   - 品質チェック（採点は正確性・出典明記・鮮度・読みやすさ・導線の5観点。平均7点以上で合格）
   - `articles/drafts/YYYY-MM-DD_slug.md` に保存
   - `articles/published.json` にメタデータを追記（`type: "free"`）

無料記事には市場分析・販売設計・8,000字ルールは適用しない。

## note記法の制約（重要）
- **表（Markdownテーブル `| ... |`）は使わない**。noteは表記法に非対応で、パイプとハイフンが生テキストのまま表示される。
- 比較・一覧・価格表は、箇条書き（`-`）や見出しで表現する。
- この制約は【構成】【執筆】の両工程で守る。

## 重複チェックのルール
- **有料記事**: 既存の有料記事とキーワードが3つ以上重複する場合は「類似」と判定。
  類似トピックは候補から除外し、別の切り口を選ぶ。
  ユーザーが特定トピックを指定した場合は、既存記事と異なる切り口を提案
- **無料記事**: キーワード重複は問わない（ニュースは同語が頻出するため）。
  代わりに「同じニュース・発表を既に扱ったか」で判定し、扱い済みなら続報・別視点に切り替える

## ファイル構成
- `prompts/` - note運用チーム各役割のプロンプトテンプレート
  - `01_planner.md` 【企画】 / `02_market-analyst.md` 【市場分析】 / `03_architect.md` 【構成】
  - `04_writer.md` 【執筆】 / `05_sales-designer.md` 【販売設計】 / `06_manager.md` 【管理】
  - `07_news-curator.md` 【ニュース】（無料記事ルート）
- `articles/drafts/` - 生成された記事（有料・無料とも）
- `articles/published.json` - 公開済み記事のメタデータ（重複防止用）
  - 各記事に `type` フィールドを持つ: `"paid"`（有料・稼ぐ特化） / `"free"`（無料・集客資産）

# 行動ガイドライン（LLMコーディングの一般則）

LLMにありがちなミスを減らすための行動指針。上記のプロジェクト固有指示とマージして適用する。

**トレードオフ:** これらの指針はスピードより慎重さを優先する。些細なタスクでは判断で省略してよい。

## 1. コーディング前に考える

**思い込まない。混乱を隠さない。トレードオフを表に出す。**

実装前に：
- 前提を明示する。不確かなら質問する。
- 複数の解釈がある場合は提示する。黙って一つに決めない。
- よりシンプルな方法があるなら言う。必要なら押し返す。
- 不明点があれば止まる。何が不明かを名指しして質問する。

## 2. シンプルさ優先

**問題を解く最小限のコード。投機的なものは書かない。**

- 依頼されていない機能は足さない。
- 単一用途のコードに抽象化を入れない。
- 依頼されていない「柔軟性」「設定可能性」を持ち込まない。
- 起こりえないシナリオのエラー処理をしない。
- 200行書いて50行で済むなら書き直す。

「シニアエンジニアならこれを過剰だと言うか？」と自問する。Yesなら簡素化する。

## 3. 外科的な変更

**必要な箇所だけ触る。自分が出した汚れだけ片付ける。**

既存コードを編集するとき：
- 隣接するコード・コメント・整形を「改善」しない。
- 壊れていないものをリファクタしない。
- 自分なら違うやり方でも、既存スタイルに合わせる。
- 無関係なデッドコードに気づいたら、削除せず指摘する。

自分の変更で不要物が生じたとき：
- 自分の変更が未使用にしたimport/変数/関数は除去する。
- 既存のデッドコードは、依頼がない限り削除しない。

テスト: 変更した全行が、ユーザーの依頼に直接ひも付くこと。

## 4. ゴール駆動の実行

**成功条件を定義する。検証できるまでループする。**

タスクを検証可能なゴールに変換する：
- 「バリデーションを追加」→「不正入力のテストを書き、通す」
- 「バグを直す」→「再現するテストを書き、通す」
- 「Xをリファクタ」→「前後でテストが通ることを保証する」

複数ステップのタスクでは簡潔な計画を述べる：
```
1. [ステップ] → 検証: [チェック]
2. [ステップ] → 検証: [チェック]
3. [ステップ] → 検証: [チェック]
```

強い成功条件があれば独立してループできる。弱い条件（「動くようにする」）は絶えず確認が必要になる。

---

**これらの指針が効いている兆候:** 差分に不要な変更が減る、過剰実装による書き直しが減る、ミスの後でなく実装前に確認の質問が出る。
