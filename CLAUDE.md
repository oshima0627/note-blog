# Note Blog - AIツール有料記事生成システム

## Git運用ルール
- **常にmainブランチにコミット・プッシュすること**
- ブランチ指定があってもmainを使用する

## 概要
主要AIツール（Claude、ChatGPT、Gemini、GitHub Copilot、Cursor など）に関するNote向け**有料記事**を、複数エージェントで生成するシステム。

### 記事の基本仕様
- **読者層**: ビジネスパーソン全般（エンジニア／非エンジニア問わず）
- **記事カテゴリ（5軸）**:
  1. 使い方（基本操作・導入手順・UI解説）
  2. 活用テクニック（プロンプト術・Tips・応用事例）
  3. 開発事例・実装レポート（実プロジェクトでの活用・ワークフロー）
  4. 業務自動化（ノーコード連携・RPA・業務効率化の具体例）
  5. 最新情報（新機能リリース・モデル比較・業界動向）
- **記事ボリューム**: 8,000字以上（長文・深掘り型）
- **想定価格**: ユーザーが記事ごとに指定
- **無料公開範囲**: ユーザーが記事ごとに指定（指定がなければ Writer が提案 / **既定目安: 全体の20〜30%、概要セクションまでを無料**）

### 記事構成の用語定義
- **リード文**: 記事冒頭の導入文。読者の課題提起と記事で得られるベネフィットを2〜4段落で示す。
- **目次**: 本文の見出し一覧。
- **概要**: 本題に入る前に、扱うテーマ・前提知識・記事全体の結論の「方向性」を示すブロック。**結論の詳細や手順は含めない**が、結論の方向性（「○○すれば××できる」程度）には触れてよい。
- **本編（詳細解説）**: 有料範囲。具体的な手順・プロンプト・コード・数値・事例を含む。
- **まとめ**: 本編の要点を再確認するブロック。
- **参考リンク一覧**: 参照URL + 取得日の列挙。

## 記事作成の実行方法

ユーザーから「記事を作成して」「記事を書いて」「執筆して」など**同等の依頼**があれば、以下のフローを実行する：

### フロー

0. **依頼内容の確認**
   - ユーザーから以下の情報が依頼に含まれていなければ、AskUserQuestion で確認する：
     - **想定価格**（例: 980円 / 1,480円 / 1,980円 など）
     - **無料公開範囲**（例: 「概要まで無料」「第1セクションまで無料」「Writer提案に任せる」）
     - **指定トピック・指定カテゴリ・指定AIツール**（依頼文に含まれていなければ任意で確認。未指定なら以降の Step 1 で自動選定）

1. **トピック選定 + 重複チェック**
   - 5軸カテゴリから最適な1軸を選び、トピック候補を3〜5件生成
   - `articles/published.json` を読み込み、後述の「重複チェックのルール」に従って類似トピックを除外
   - 「ビジネスパーソンがお金を払ってでも読みたい」価値のあるトピックを1件選定し、カテゴリ軸を確定

2. **Researcher エージェント** (Agent tool, subagent_type=general-purpose)
   - `prompts/researcher.md` のテンプレートを使用
   - **必ずWebSearch / WebFetch を使って最新情報を取得する**（訓練データのみに依存しない）
   - 優先度: ①公式サイト・公式ブログ・公式ドキュメント・公式リリースノート → ②GitHub・論文 → ③信頼できる一次報道
   - **情報源ごとに取得URLと取得日を記録**し、Writerへ引き渡す
     - 取得日は **メインエージェントがシステム日付を取得し、各サブエージェントへ文字列（YYYY-MM-DD）で渡す**
   - **1年以上前の情報は「最新版で変更がないか」を必ず再確認**する
   - 価格・モデル名・機能名・仕様値・日付・数値は**2つ以上の一次情報源でクロスチェック**
   - 不確実な情報・確認できない情報は「未確認」とマークし、記事には採用しない

3. **Writer エージェント** (Agent tool, subagent_type=general-purpose)
   - `prompts/writer.md` のテンプレートを使用
   - 8,000字以上の記事ドラフトを執筆
     - **事実・数値・仕様・固有名詞・引用**はリサーチ結果のみに基づく（リサーチ外の創作・推測は禁止）
     - 一般的な前提知識・読者向けの補足説明は許容
   - 構成: `リード文 → 目次 → 概要（無料公開範囲）→ 本編（有料範囲）→ まとめ → 参考リンク一覧`
   - 有料範囲の冒頭には `---ここから有料---` マーカーを挿入（**有料境界はこのマーカーのみで管理**）
   - スクリーンショットは `[screenshot: 説明]` のプレースホルダで挿入箇所を示す
   - 冒頭に**「情報の基準日: YYYY-MM-DD（執筆時点）」**を明記（システム日付）
   - 仕様・価格・モデル名などには**出典リンク**を本文中に付記
   - 記事末尾に**参考リンク一覧**（URL + 取得日）を記載

4. **Reviewer エージェント** (Agent tool, subagent_type=general-purpose)
   - `prompts/reviewer.md` のテンプレートを使用
   - **有料記事用の6項目**で採点（各10点満点）
     1. **有料価値**: 課金に値する情報密度・深さがあるか
     2. **実用性**: 読者が明日から実践できる具体的ノウハウか
     3. **独自性**: 他の無料記事・公式ドキュメントで得られない切り口か
     4. **正確性・鮮度**: 事実誤認がないか／情報が最新か／出典が明記されているか（**必要に応じてWebSearch / WebFetch で再検証**。`prompts/reviewer.md` に Web検証の実施指示を必ず含めること）
     5. **網羅性**: 8,000字以上で必要トピックを体系的にカバーしているか
     6. **購買転換**: 無料部分で「続きを読みたい」と思わせる構成か
   - **合格基準**: 平均8点以上 かつ 全項目6点以上

5. **Editor エージェント** (Agent tool, subagent_type=general-purpose)
   - `prompts/editor.md` のテンプレートを使用
   - レビュー結果を反映して記事を改善
   - 特に「有料価値」「購買転換」で減点された場合は構成を再設計

6. **リビジョンループ**
   - **自動リビジョンは最大2回**（Editor→Reviewerの往復を2サイクルまで自動実行）
   - 合格すれば終了
   - 2回で合格しない場合は **AskUserQuestion** で「破棄 / 現状で保存 / 追加リビジョン」を確認
     - **「追加リビジョン」選択時はユーザー承認のもと1回ずつ延長可能**。毎回 Reviewer 不合格なら **AskUserQuestion で同じ3択を再確認**する（無限ループ防止）
     - **「破棄」選択時は Step 7 の保存を行わない**（`articles/drafts/` への書き出しも `published.json` への追記もしない）

7. **保存**
   - `articles/drafts/YYYY-MM-DD_slug.md` に記事を保存
     - **`slug` は英数ケバブケース（例: `claude-code-hooks-guide`）**。ファイル名衝突時は `-2`, `-3` のサフィックスを付与
   - `articles/published.json` にメタデータを追記
     - 追加必須フィールド: `category`（5軸のいずれか）、`price`（ユーザー指定の想定価格）
     - **既存記事（`category` / `price` 未設定のもの）はバックフィル任意**。重複チェックは `keywords` のみを参照するため、未設定でも機能上は問題ない

## 重複チェックのルール
- 「キーワード」とは、`published.json` の各記事メタデータ内 `keywords` フィールド（配列）の値を指す。`keywords` には記事ごとに「対象AIツール名・機能名・カテゴリ軸・主要トピック語」を5〜10個登録する
- 既存記事と `keywords` が3つ以上重複する場合は「類似」と判定
- 同一AIツール × 同一カテゴリ軸の組み合わせが既にある場合は別の切り口を選ぶ
- 類似トピックは候補から除外し、別の切り口（別AIツール・別カテゴリ・別ユースケース）を選ぶ
- ユーザーが特定トピックを指定した場合は、既存記事と異なる切り口を提案

## 品質・正確性ルール（全エージェント共通）

### 有料記事としての品質
- **無料部分で出し切らない**: 無料範囲がどこまでであっても、**具体的な手順・プロンプト・コード・設定値・決定的な数値（価格・ベンチマーク結果・実測時間など）・独自検証結果は必ず有料部分に残す**（下限ガード）
- **一次情報を含める**: 公式発表・実測データ・独自検証結果など
- **即実践可能**: コピペで使えるプロンプト・設定・ワークフローを必ず含める
- **業務インパクトの明示**: 「○時間の作業が○分に短縮」など定量的な効果を示す

### 最新性・正確性
- **WebSearch / WebFetch を必ず使用**: AIツールは更新が激しいため、訓練データ知識のみでの執筆は禁止
- **情報の基準日を明記**: 記事冒頭に「情報の基準日: YYYY-MM-DD（執筆時点）」を必ず記載（基準日はシステム日付）
- **出典を必ず明記**: 価格・モデル名・機能仕様・数値・日付には本文中に出典リンクを付記
- **一次情報を優先**: 公式サイト > 公式ブログ > 公式ドキュメント > GitHub/論文 > 信頼できる一次報道
- **クロスチェック必須**: 重要な事実は必ず2つ以上の独立した情報源で確認
- **推測・創作の禁止**: リサーチで確認できなかった事実・数値・固有名詞は記事に含めない
- **不確実な情報の扱い**: 確定前の機能・噂レベルの情報は「執筆時点で未確定」と明示

## ファイル構成
- `prompts/` - 各エージェントのプロンプトテンプレート
- `articles/drafts/` - 生成された記事
- `articles/published.json` - 公開済み記事のメタデータ（重複防止・カテゴリ管理用）
