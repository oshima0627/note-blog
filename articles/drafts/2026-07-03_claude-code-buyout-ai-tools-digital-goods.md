# Claude Codeで「買い切りAIツール」を量産して月5万円：Gumroad/BOOTHで売るデジタル物販ルート

サブスクの保守や解約対応に、もう疲れていませんか。この記事は、Claude Codeで1〜3日で作れる小さなツールを「買い切り（都度課金）」で売り、複数本の合算で月3万〜10万円を狙うルートを、手数料の実額から逆算して設計します。ただし先に正直にお伝えします。この金額はプラットフォーム手数料から計算した目安であって、保証ではありません。実売上の第三者検証済み事例はWebSearchでも確認できませんでした。だからこそ本記事は、唯一確実な事実である「手数料」を土台に、あなた自身が価格×本数で数字を組み直せる「計算の型」をお渡しします。感覚ではなく計算で、稼げるかを自分で判断できるようになります。

## この記事の対象読者と、狙える収益の目安（先に結論）

まず結論からお伝えします。狙う金額は月3万〜10万円が目安です。これは「買い切り1,500円のツールを、手数料を差し引いた手取りで積み上げ、月合計35〜70本ほど売れた場合」の計算上の数字です。保証された収益ではありません。実際の販売数は商品力・集客・時期に大きく左右されます。

この記事は、次のような方に向いています。

- サブスク運用（解約対応・継続保守・サポート対応）の負担なく、作って売り切りたい方
- 個人開発したツールの「売る場所」と「決済・ライセンス配布」の具体的な手順を知りたい方
- 月5万円という金額を、どの価格×何本×何プラットフォームで組むかの逆算で理解したい方
- Chrome拡張・デスクトップ小物・スクリプトなど、小規模な成果物を換金する導線を知りたい方
- 海外（Gumroad）と国内（BOOTH）で、いくら手数料が引かれ手取りがいくら残るか比較したい方

正直にお伝えします。個人開発者が実際に月5万〜10万円を売り上げた監査済みの記録は、調べた範囲では確認できませんでした。世に出ている作例は、本人が申告した試算値・理論値です。ですから本記事では「実績」を断定しません。代わりに、一次情報で裏の取れる「手数料・登録料」という確実な事実だけを土台にします。読み終えたとき、あなたが手にするのは3つです。全体像、手取り計算の型、そして6つの行動ステップです。この3つがあれば、自分の数字で判断できます。まずは、なぜ今サブスクではなく買い切りなのかから見ていきましょう。

## なぜ今「サブスク」ではなく「買い切りAIツール」なのか

近年、個人開発でも「サブスク（月額課金）」が推奨されてきました。継続収入が積み上がるからです。ただし、サブスクには見えにくいコストがあります。

サブスクは、売って終わりではありません。契約が続く限り、保守が続きます。ライブラリの更新、APIの仕様変更への追従、障害対応が発生します。解約手続きへの対応も要ります。問い合わせ対応も、契約者が増えるほど積み上がります。つまり「稼働し続ける義務」を負う設計です。副業で始めるには、この継続負担が重くのしかかります。

買い切り（都度課金）は、この構造が逆です。一度売れば、その取引は完結します。追加の課金は発生しません。だから商品説明に「買い切り・追加課金なし」と明記できます。サブスク疲れの層には、これ自体が刺さる訴求になります。

そしてClaude Codeは、この買い切りモデルと相性が良いのです。Claude Codeは、実装・テスト・ドキュメント作成までを対話で進められる開発ツールです。単機能に絞った小さなツールなら、1〜3日という短い粒度で形にできます。1本の完成コストが低いほど、本数を増やしやすくなります。

戦略はシンプルです。単機能ツールを量産し、「本数 × 手取り」で積み上げます。1本が生む月数千円は小さく見えます。しかし、新規1本を追加するコストが低いため、隣接するニーズへ横展開しやすいのです。1本目で得た知見を2本目・3本目に流用できます。売れ筋が見つかれば、その周辺で商品を増やせます。売れないものは撤退すればよいだけです。在庫を持たないデジタル物販だからこそ、この試行錯誤のコストが低く済みます。

## 収益逆算：価格×本数×手取りで「月5万円」を分解する

「月5万円」という数字を、感覚ではなく計算で分解します。ここが本記事の核です。

基本式はこうです。

**手取り単価 × 販売本数 = 月間手取り**

まず手取り単価を決めます。価格帯は1,000〜3,000円を想定します。デジタルの小物ツールとして、買いやすく、かつ手数料負けしない現実的なレンジです。ここでは1,500円を例にします。

次に、手数料を差し引いた手取りを出します。詳細は次のセクションで比較しますが、BOOTHの場合、1,500円の商品なら手数料は約129円です。手取りは約1,371円になります（BOOTH公式ヘルプ、2026年7月時点。最新は公式で要確認）。

この手取りで月5万円に届く本数を逆算します。

- 5万円 ÷ 手取り約1,371円 ≒ 約37本
- 手取り率や価格帯の幅を見て、月35〜70本のレンジで考える

つまり、1,500円のツールが月に35〜70本売れれば、5万円前後の手取りになる計算です。1日あたりに直すと、およそ1〜2本です。

ここで大事な発想の転換があります。この「35〜70本」を、1商品だけで達成する必要はありません。「1本あたり月に数本 × 複数の商品」で合算するのです。例えば、月に数千円しか売れないツールでも、それが5本あれば合算で数万円になります。1本のヒットに頼るのではなく、複数の小さな流れを束ねる設計です。これが「量産して積み上げる」の中身です。

繰り返しますが、この本数はあくまで計算上の目標です。実際に何本売れるかは、商品の有用性、集客の届き方、そして時期に依存します。誰でも自動的に届く数字ではありません。ただ、こうして式に分解しておけば、「価格を上げるか」「本数を増やすか」「手取り率の高い売り場を選ぶか」という打ち手を、自分で選べるようになります。

## Gumroad と BOOTH：手数料と手取りを実額で比較する

さきほどの逆算式で使った「手取り」は、どの売り場を選ぶかで変わります。ここで実額を押さえておけば、あとの価格設定（STEP4）で迷いません。売り場選びは、手取りに直結します。ここでは主要な2つを、実額で比較します。すべて2026年7月時点の情報です。手数料は改定されることがあるため、最新は各公式で必ず確認してください。

**BOOTH（国内）**

BOOTHはpixivが運営する国内の販売プラットフォームです。ダウンロード商品のサービス利用料は、販売額×5.6%+45円です。この固定額45円は、2025年10月28日に22円から45円へ改定されました。

- 1,500円の商品の場合：手数料は約129円（1,500円×5.6%=84円 + 45円）
- 手取りは約1,371円

売上を受け取る際の振込手数料もあります。

- 銀行振込：3万円未満は200円、3万円以上は300円（いずれも税込）
- PayPal・Wiseでの受け取り：無料

国内ユーザーが買いやすく、固定手数料が低いのが強みです。少額商品でも手取りが残りやすい構造です。

**Gumroad（海外）**

Gumroadは海外発の販売プラットフォームです。直接販売（自分の販売ページ経由）の手数料は、10%+$0.50/取引です。月額固定費はありません。

- Gumroadの「Discover」（プラットフォーム内の発見機能）経由で売れた場合は、手数料が30%になります
- 1,500円（約$10）の商品を直接販売した場合、手取りは概ね販売額の8割前後です

Gumroadは2025年1月より、Merchant of Record（MoR、販売事業者）として各国の税処理を代行しています。海外向けに売るときの税の扱いを肩代わりしてくれる仕組みです。ただし注意点があります。返金が発生しても、手数料は返ってきません。返金対応をする場合、その取引の手数料分は戻らないと理解しておいてください。

**使い分けの考え方**

- 国内の顧客が中心なら、固定手数料の低いBOOTHが有利です
- 海外向けや税処理を任せたいなら、MoRのGumroadが選択肢になります
- 最初から両方に手を広げる必要はありません。まずどちらか片方で、手取り計算を回してみるのが現実的です

いずれの数字も2026年7月時点のものです。手数料率や固定額は改定される可能性があるため、出品前に必ず公式ヘルプで最新を確認してください。

## Chrome拡張の二段構え：ストア無料公開で集客→外部ライセンスで売る

作るツールがChrome拡張機能なら、少し特殊な設計が必要です。ここを知らずに始めると、売る段階で詰まります。

まず、Chrome Web Storeで拡張を公開するには、デベロッパー登録料が必要です。一回$5です。これは初回のみの支払いで、審査もあります（出典：developer.chrome.com、2026年7月時点）。

重要なのは課金の仕組みです。Chrome Web Storeは、ストア内での直接課金を廃止済みです。つまり「ストアで拡張を有料販売する」ことはできません。課金したい場合は、外部でライセンスを販売する設計が必要になります。

そこで有効なのが「二段構え」です。

- ストアには無料版を公開する。これを集客装置として使う
- 上位機能や制限解除は、外部（Gumroadなど）でライセンスキーを販売して提供する

無料版でユーザーに見つけてもらい、価値を体感してもらいます。その上で、もっと使いたい人に有料ライセンスを売るのです。ストアの検索流入を入口に、外部の売り場へつなぐ流れです。

ただしリスクもあります。拡張はストアの審査を通す必要があり、リジェクト（公開却下）される可能性があります。ポリシー変更によって、公開中の拡張が停止されることもあります。1つのプラットフォームに依存すると、そこが止まった瞬間に導線が消えます。だからこそ、拡張だけに頼らず、後述する複数の入口を持っておくことが大切です。

## 6つの行動ステップ：ネタ決めから月5万円まで

ここからが実務です。ネタ決めから積み上げまでを、6つのステップに分けて解説します。この6ステップは、STEP1〜3が「作って売り場に並べる」まで、STEP4〜6が「売れる形に整えて積み上げる」まで、と2つの山に分かれます。ひとつずつなら難しくありません。この順番で進めれば、迷わず動けます。

**STEP1 売れる小物のネタを3本決める**

起点は「毎回手作業でやっている面倒」です。自分か身近な人が、繰り返している定型作業を思い出してください。そこにツール化の種があります。

Claude Codeで1〜3日で作れる粒度に絞ります。候補の型はこうです。

- 特定サイト向けのChrome拡張（そのサイトでの定型操作を自動化）
- CSV整形などのデスクトップ小物
- 定型作業を片付けるスクリプト
- Notion・Slackと連携する小さなツール

コツは「ニッチで検索されやすい単機能」を選ぶことです。何でもできる万能ツールは、作るのも売るのも難しくなります。むしろ「あの作業だけを、確実に速くする」1点特化のほうが、困っている人に届きます。この段階で3案を書き出します。「作れる」「売り場がある」「誰かが困っている」の3つが交わるものが、初回商品の候補です。

**STEP2 Claude CodeでMVPを作る**

MVP（必要最小限の製品）を作ります。ここで欲張らないことが重要です。機能は1つに絞ります。買った人が、すぐに使える状態を目指します。

Claude Codeで、実装・テスト・README（使い方説明）・スクリーンショットまで用意します。買い切り商品では、購入後のサポートを軽くしたいはずです。だからこそ、ドキュメントとスクショで「説明なしでも使える」状態に仕上げます。

つまずきやすいのが依存関係です。使っているライブラリのライセンス（後述）と、動作環境を手元で必ず確認してください。「自分の環境では動くが、買った人の環境で動かない」は、返金やクレームの原因になります。動作条件は明確にしておきます。このステップの完了地点は、配布可能なzipや拡張パッケージと、購入者向けドキュメントが揃った状態です。

**STEP3 売り場を用意する（BOOTH＋Gumroad）**

売り場を開設します。前述の通り、まず片方から始めるのが現実的です。

- 国内向け：BOOTH。pixivアカウントで開設できます。ダウンロード商品として出品します。手数料は5.6%+45円です
- 海外向け：Gumroad。直接販売の手数料は10%+$0.50です。MoRとして税処理を代行してくれます

どちらも初期登録・出品は無料です。前金は要りません。手数料は売れた分から差し引かれる方式です。だから、売れる前にお金を失うことはありません。このステップの完了地点は、決済とダウンロード配布が自動化された販売ページが公開された状態です。ここまで来れば、あとは24時間、勝手に売り場が動きます。

**STEP4 価格と買い切りの見せ方を決める**

価格を決めます。レンジは1,000〜3,000円です。目安はこうです。

- 機能単体：1,000〜1,500円
- セット販売やサポート付き：2,000〜3,000円

価格を決めるときは、必ず手取りを意識します。手数料を引いた後にいくら残るかを、STEP3の手数料で計算してから値付けします。

そして「買い切り＝追加課金なし」を、商品説明ではっきり打ち出します。これがサブスク疲れの層への訴求点です。あわせて、返金の条件とサポートの範囲を明記します。「どこまで対応し、どこからは対応しないか」を先に書くことで、後のトラブルを防げます。このステップの完了地点は、手取りを意識した価格と、誤解のない商品説明が完成した状態です。

**STEP5 1本目を売り導線を作る**

売り場を作っただけでは、まず売れません。人が来ないからです。ここで集客の導線を作ります。

- Xで、開発過程と使い方のGIF（短い動画）を発信する
- Zenn・note・Qiitaで、「作った記録」の記事を書き、そこから商品ページへ誘導する

「作りました、買ってください」だけでは弱いのです。作る過程を見せると、興味と信頼が生まれます。技術的な知見を無料で共有し、その延長で「同じ悩みを解決するツールがあります」と案内する流れです。

Chrome拡張なら、前述の二段構えが効きます。ストアの無料版で集客し、上位機能を外部ライセンスで売ります。無料版が、そのまま広告の役割を果たします。このステップの完了地点は「無料の技術発信 → 有料ツール購入」という、再現性のある導線ができた状態です。

**STEP6 本数を積み上げて月5万円に寄せる**

1本目で手応えをつかんだら、量産に入ります。売れ筋が見つかったら、その隣接ニーズで2本目・3本目を作ります。1本目で作った実装や販売ページの型を流用できるため、新規1本の追加コストは低く抑えられます。

思い出してください。目標は「1本あたり月数千円 × 複数本の合算」です。1本のヒットを狙うのではなく、複数の小さな流れを束ねます。そして、売れない商品は潔く撤退します。デジタル物販は在庫を持たないので、撤退コストがほぼゼロです。だから、たくさん試して、当たりを残す戦略が取れます。この積み上げの先に、月3万〜10万円水準を狙える状態があります。ただし、これは目標であって保証ではありません。到達には、商品力と継続的な発信が要ります。

## 集客チャネルの組み方：無料の技術発信から有料ツールへ

STEP5でも触れましたが、集客は成否を分ける要素なので、あらためて整理します。売り場の手数料より、集客の設計のほうが、実は収益を大きく左右します。

軸になるチャネルは3種類です。

- **X**：開発過程やデモGIFをリアルタイムに発信する場所。瞬発力があり、拡散が起きやすい
- **Zenn・Qiita・note**：作った記録を記事として残す場所。検索から長く読まれ、ストックとして効く
- **GitHub**：READMEやコードを通じて信頼を得る場所。「ちゃんと作れる人」という裏付けになる

考え方はこうです。無料の技術発信が、信頼と動線を作ります。いきなり商品を売るのではなく、まず役立つ情報を無料で出します。その情報にたどり着いた人が、書き手を信頼し、その延長で商品を知る流れです。

Chrome拡張の無料版も、この文脈では集客装置です。無料で使ってもらい、価値を体感してもらった上で、有料版へ案内します。

そして重要なのが、入口を複数持つことです。Xだけ、ストアだけ、という1本足打法は危険です。プラットフォームの仕様変更やアカウント停止が起きれば、集客がゼロになります。X・記事プラットフォーム・GitHub・ストアと、複数の入口を分散させておくことで、どれか1つが止まっても導線が残ります。

## 必要なスキルと初期コスト：始める前の現実確認

始める前に、必要なものを現実的に確認します。過大な期待も、過剰な不安も持たずに判断するためです。

**必要なスキル**

- Claude Codeで小規模ツールを実装し、パッケージ化する力
- 配布形態（zip・インストーラ・拡張・ライセンスキー）を用意できる基礎
- 商品説明・スクショ・デモを作る、マーケティング表現の力
- 手数料を差し引いた手取りを計算し、価格を設定する力
- X・Zennなどでの技術発信による集客
- 利用規約・OSSライセンス・返金対応についての基礎理解

すべてを最初から完璧に備える必要はありません。1本作る過程で、少しずつ身につきます。

**初期コスト（2026年7月時点、最新は公式で要確認）**

- Chrome Web Storeデベロッパー登録料：一回$5（拡張を出す場合のみ）

- BOOTH・Gumroadの初期登録・出品：無料

金銭的な初期投資は、かなり小さく抑えられます。拡張を出さないなら、実質的な固定費は開発ツールの費用が中心です。

**時間**

1本目のツール実装と出品で、数日〜2週間が目安です。前述の通り、手数料は売れた分から差し引かれる方式で、前金はありません。だから金銭リスクは限定的です。むしろ主な投資は「時間」です。この時間を、回収できるかどうかを見極めながら進めます。

## リスク・注意点：売る前に詰まる関門と回避策

最後に、正直なリスクを共有します。ここを飛ばすと、売る直前や売った後で詰まります。

**収益は保証されません**。月3万〜10万円は、手数料から逆算した目安です。実際の販売数は、商品力・集客・時期・季節性に依存します。実売上の第三者検証済み事例は確認できていません。「やれば必ず稼げる」ものではないと理解してください。

**規約とプラットフォーム依存のリスク**があります。禁止されている商材を売ればアカウントが停止されます。規約は変更されます。Chrome拡張は、ポリシー変更で公開停止になる可能性があります。回避策は、1つのプラットフォームに依存しないことです。売り場と集客の入口を分散させます。

**陳腐化と競合過多のリスク**があります。単機能ツールは模倣されやすいものです。また、APIやブラウザの仕様変更で、突然動かなくなることがあります。これは保守リスクです。回避策は、動作保証範囲と非保証範囲を、商品説明にあらかじめ明記することです。「この環境で動作を確認、将来の仕様変更は非保証」と書いておけば、期待値のズレを防げます。

**OSSライセンスと著作権の注意**があります。依存しているOSS（オープンソースソフトウェア）のライセンス条件を守ってください。MITやGPLなど、種類によって条件が違います。生成物に第三者の権利物を含めないことも重要です。

**税の注意**があります。日本では、副業所得が一定額を超えると確定申告が必要です。海外Gumroadの売上や、為替差も所得に含まれます。申告の要否は、税務の公式情報で各自確認してください。売れ始めたら、早めに記録を残す習慣をつけると安心です。

これらのリスクは、事前に知っていれば、多くが回避または軽減できます。知らずにぶつかると痛手になる関門を、先回りして潰しておきましょう。
