# Claude Codeで4.6億行を監査：アルバータ州政府の実例

2026年7月7日時点の情報です（元発表は2026年7月6日）。料金・仕様は変わりやすいため、最新は各公式でご確認ください。

## この記事でわかること

- カナダ・アルバータ州政府がClaude Codeでコード全体をセキュリティ監査した事例の概要

- 個人開発者・エンジニアが自分の開発に取り込める考え方

## 何があったのか（概要）

2026年7月6日、Anthropicはカナダ・アルバータ州政府の事例を公開しました。
州の技術革新省（Ministry of Technology and Innovation）が、
Claude Code（OpusとSonnet）を使って自組織のコードを点検した取り組みです。

発表によると、約50体のエージェントが自律的かつ並列に動き、
**4億6,600万行のコードを約20時間で評価**したとされています。
対象は**1,280のアプリケーションと3,400のコードリポジトリ**に及びます。
これらのシステムには税務記録・政府調達データ・社会福祉のケースファイルなど、
機微な情報が含まれると説明されています。

取り組み自体は2025年から続いているもので、今回はその成果がまとめて公表された形です。

## 具体的にどう進めたのか

発表では、進め方が「スキャン → 修正 → レビュー」の流れで説明されています。

**1. スキャン（2段構え）**
Claude Codeは2段階のルーチンで動いたとされています。
まず各リポジトリを「ルールエンジン」で走査し、既知のパターン（脆弱性の型）を洗い出します。
次に、その検知結果をレビューし、指摘ごとに該当ファイルと行番号を明示したと説明されています。

**2. 修正（テストがなければテストから書く）**
脆弱性が見つかった箇所では、Claude Codeが修正案の生成・テスト・ビルドまで行えたとされています。
パッチの安全性を確認する自動テストが無いシステムでは、
**まずテストを書いてから修正した**と説明されています。
古すぎる・複雑すぎて効率的に直せないコードは、
より新しく保守しやすい言語で作り直し、一部は4〜5日で完了したとされています。

**3. 人によるレビュー**
「どのパッチも、出荷前にチームがレビューし承認した」と明記されています。
つまり最終判断は人が握る運用です。

## Claude Codeユーザーへの影響（筆者の見方）

ここからは筆者の見方です。この事例が示すのは、次の3点だと考えられます。

- **並列エージェントは「規模の道具」になる**: 1体で順番に見るのではなく、
  リポジトリ単位で多数のエージェントに割り振る発想は、個人の大規模リポジトリにも応用できます。
- **「機械の網 → AIの精査」の2段構えが効く**: いきなりLLMに全部読ませるのではなく、
  ルールベースで候補を絞ってからClaudeに精査させる設計は、コストと精度の両面で理にかなっています。
- **テストが無いなら書かせてから直す**: 「テストで安全を確認してから修正」という順番は、
  レガシー改修で事故を防ぐ堅実な型で、個人開発でも真似しやすい考え方です。

一方で、これは高度な体制と人によるレビューを前提にした政府の取り組みです。
「AIに任せれば自動でセキュアになる」という話ではありません。
最終承認を人が行っている点は、そのまま踏襲したい部分です。

なお発表によると、アルバータ州は他の政府機関が学べるよう、
一連の取り組みをまとめた技術ホワイトペーパー群を公開したとされています。
公的機関がAIによるセキュリティ点検の手順を公開する動きは、
同種の取り組みを検討する組織にとって参考材料になりそうです。

## 個人開発で試すなら（筆者の提案）

ここも筆者の提案で、事例そのものではありません。
同じ発想は、規模を小さくすれば個人のリポジトリでも試せます。

- **範囲を絞る**: いきなり全体ではなく、認証・入力処理・外部通信など
  リスクの高いディレクトリから始めると、指摘の精度を確認しやすくなります。
- **「指摘→ファイルと行番号」を必ず出させる**: 事例と同じく、
  該当箇所を明示させると、誤検知かどうかを人が短時間で判断できます。
- **テストが無い箇所は先にテストを書かせる**: 修正の前に振る舞いを固定しておくと、
  パッチで別の不具合を生むリスクを下げられます。
- **修正は必ず人がレビューしてから反映する**: 自動生成のパッチを鵜呑みにせず、
  差分を読んで承認する運用を崩さないことが大切です。

派手な自動化よりも、「絞る・根拠を出させる・人が承認する」という地味な型が、
そのまま実務の安全性につながる——というのが、この事例から読み取れる教訓だと考えます。
