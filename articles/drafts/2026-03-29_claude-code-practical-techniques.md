# Claude Codeで開発効率を劇的に上げる5つの実践テクニック

**対象読者：** 日常的にターミナルで開発しているエンジニア
**この記事で得られること：** Claude Codeの導入から実践的な活用法まで、明日から使えるテクニック

---

「AIコーディングツール、結局どれを使えばいいの？」

GitHub Copilot、Cursor、Aider......選択肢が増える一方で、どれも帯に短したすきに長しと感じていませんか。エディタ内の補完だけでは物足りない。かといってエディタごと乗り換えるのは腰が重い。そんな方にこそ試してほしいのが、Anthropic公式のCLIツール「Claude Code」です。

本記事では、Claude Codeを使って開発ワークフローを効率化する5つの実践テクニックを紹介します。

---

## Claude Codeとは何か

Claude Codeは、Anthropicが提供するターミナルベースのAIエージェントです。いわゆるコード補完ツールではありません。ターミナル上で対話しながら、ファイルの読み書き、検索、Git操作、シェルコマンド実行などをAIが自律的に判断して行います。

特徴を簡潔にまとめると、以下の通りです。

- **エージェント型：** 指示を出すだけで、必要なファイルを探し、編集し、テストまで実行する
- **コードベース全体を理解：** プロジェクト構造を把握した上で回答・作業する
- **エディタ非依存：** VS CodeでもVimでもEmacsでも、ターミナルさえあれば使える
- **MCP連携：** 外部ツールとの統合が可能（後述）

## セットアップ：5分で始められる

インストールは非常にシンプルです。

```bash
npm install -g @anthropic-ai/claude-code
```

インストール後、プロジェクトディレクトリで `claude` と入力するだけで起動します。

```bash
cd your-project
claude
```

初回起動時にAnthropicアカウントとの認証が求められます。指示に従って認証を済ませれば、すぐに使い始められます。

### 料金体系を把握しておく

Claude Codeの料金体系は2種類あります。

- **API従量課金：** Anthropic APIのトークン使用量に応じて課金される。短時間の利用や試用に向く
- **Max Plan（月額定額）：** Claude Pro/Maxプランに含まれる形で利用可能。日常的に使うならこちらの方がコストを管理しやすい

どちらを選ぶかはプロジェクトの規模と利用頻度次第です。筆者の感覚では、1日に30分以上使うならMax Planの方が割安になるケースが多いです。

## テクニック1：CLAUDE.mdでプロジェクト理解度を最大化

Claude Codeの出力品質を大きく左右するのが `CLAUDE.md` ファイルです。プロジェクトルートに配置するMarkdownファイルで、AIに対するプロジェクト固有の指示書として機能します。

手動で書くこともできますが、まずは自動生成がおすすめです。

```bash
claude /init
```

このコマンドでプロジェクト構成が分析され、雛形が自動生成されます。生成された `CLAUDE.md` に以下の情報を追記しましょう。

- **技術スタック：** 使用言語、フレームワーク、主要ライブラリ
- **コーディング規約：** 命名規則、ディレクトリ構成のルール
- **アーキテクチャ方針：** レイヤー構成、依存関係のルール
- **禁止事項：** 使ってほしくないパターンや非推奨のAPI

### 実践Tips：CLAUDE.mdを育てるコツ

自動生成されたままでは効果は限定的です。以下のように段階的に育てていくのが効果的です。

```markdown
# プロジェクト概要
TypeScript + Next.js 14（App Router）のECサイト。

# コーディング規約
- コンポーネントは関数コンポーネント + Arrow Function で統一
- 状態管理はReact Server Componentsを優先し、クライアント状態が必要な場合のみzustandを使用
- APIレスポンスの型はzodスキーマから生成する（手書き禁止）

# ディレクトリ構成
- src/app/ ... ルーティングとページコンポーネント
- src/features/ ... 機能単位のモジュール（各featureにcomponents/, hooks/, api/を持つ）
- src/shared/ ... 共通コンポーネントとユーティリティ

# やってはいけないこと
- any型の使用（unknown + 型ガードで対応すること）
- default exportの使用（named exportに統一）
- src/shared/ から src/features/ への依存（逆方向のみ許可）
```

ポイントは「やってはいけないこと」の記述です。Claude Codeはやるべきことだけでなく、避けるべきことを明示すると出力の精度が顕著に上がります。実際、筆者のプロジェクトでは `any` 型の混入がCLAUDE.mdに禁止事項を追記しただけでほぼゼロになりました。

`CLAUDE.md` の品質が高いほど、Claude Codeの出力精度は上がります。チーム開発では、このファイルをリポジトリにコミットしておくと全員が同じ品質のAI支援を受けられます。

## テクニック2：大規模リファクタリングをAIに任せる

Claude Codeが真価を発揮するのは、複数ファイルにまたがる変更です。ここでは具体的なプロンプト例を見てみましょう。

### プロンプト例：エラーハンドリングの統一

```
src/api/ 配下の全APIハンドラーを確認して、エラーハンドリングを以下のパターンに統一してください。

統一パターン：
- すべてのハンドラーで try-catch を使用
- catchブロックでは AppError 型に変換して返す
- HTTPステータスコードは AppError.statusCode から取得
- エラーログは logger.error() で出力（console.error は使わない）

既存の src/api/users.ts が正しいパターンの参考例です。
```

このプロンプトを渡すと、Claude Codeは以下を自律的に実行します。

1. `src/api/users.ts` を読んで「正しいパターン」を把握
2. `src/api/` 配下の全ファイルを走査し、現状のエラーハンドリングを分析
3. パターンが異なるファイルを特定し、一貫性のある変更を適用
4. 変更内容のサマリーを提示

### ハマりポイント：一度に変更するファイル数に注意

筆者の経験では、1回のプロンプトで変更対象が20ファイルを超えると、後半のファイルで品質が落ちる傾向があります。対象が多い場合はディレクトリ単位で分割して依頼する方が確実です。

```
まず src/api/auth/ 配下のファイルだけ修正してください。完了したら次のディレクトリに進みます。
```

また、変更内容は必ず目視で確認してください。Claude Codeはファイル編集やコマンド実行の前にユーザーの承認を求めるため、意図しない変更はそこで防げます（権限モデルについては後述）。

## テクニック3：Git/GitHub連携でPRワークフローを自動化

Git操作はClaude Codeの得意分野です。

### コミットメッセージの自動生成

```bash
# 変更内容を分析して適切なコミットメッセージを自動生成
claude /commit
```

`/commit` はステージング済みの変更を分析し、Conventional Commits形式などプロジェクトの慣例に沿ったメッセージを生成してくれます。

### PRレビューの実行

PRレビューは自然言語で依頼します。

```
gh pr diff 42 を確認して、以下の観点でレビューしてください。
- ロジックの不整合やバグの可能性
- コーディング規約との乖離
- パフォーマンス上の懸念
- テスト不足の指摘
```

または、PRのURLを直接渡す方法もあります。

```
https://github.com/your-org/your-repo/pull/42 をレビューして
```

Claude Codeは `gh` コマンドを内部で実行してPRの差分を取得し、レビューコメントを生成します。人間のレビューを置き換えるものではありませんが、一次レビューとして活用すると、人間のレビュアーがロジックや設計判断に集中できるようになります。

### 実践Tips：レビュー精度を上げるコツ

CLAUDE.mdにコーディング規約が記述されていると、レビュー精度が格段に上がります。「規約に沿っているか」を機械的にチェックする部分はAIに任せ、人間は設計判断やビジネスロジックの妥当性に集中する、という分担が効果的です。

## テクニック4：テスト自動生成でカバレッジ向上

「テストを書く時間がない」は多くのチームが抱える課題です。Claude Codeにテスト生成を依頼する際は、具体的な指示を添えると品質が上がります。

### 基本的なプロンプト例

```
src/services/userService.ts のユニットテストを作成して。
既存の src/services/__tests__/authService.test.ts のスタイルに合わせてください。
```

### より精度の高いプロンプト例

```
src/services/userService.ts のユニットテストを作成して。

要件：
- 既存の src/services/__tests__/authService.test.ts のスタイルに合わせる
- 外部依存（DatabaseClient, EmailService）はモックする
- 正常系：ユーザー作成、取得、更新、削除の各操作
- 異常系：重複メール、存在しないユーザーID、バリデーションエラー
- エッジケース：空文字列、極端に長い入力、SQLインジェクション的な文字列
```

### 生成されたテストのレビューポイント

AIが生成したテストは、以下の点を重点的に確認してください。

- **モックの妥当性：** 実際の依存先の挙動と乖離していないか。特に異常系のモックが現実的な値を返しているか
- **アサーションの甘さ：** `toBeTruthy()` のような曖昧なアサーションで済ませていないか。具体的な値で検証しているか
- **テスト間の独立性：** テストの実行順序に依存していないか。`beforeEach` でのリセットが適切か

筆者の経験では、生成されたテストの8割はそのまま使えますが、残り2割はモックの挙動やエッジケースの期待値に手動の調整が必要でした。特に、非同期処理のタイミングに関するテストは人間の確認が欠かせません。

## テクニック5：MCP連携で開発ワークフロー統合

MCP（Model Context Protocol）は、AIと外部ツールを接続するための標準プロトコルです。Claude CodeはこのMCPサーバーとの連携に対応しています。

### MCPサーバーの設定例

プロジェクトルートの `.claude/` ディレクトリ、またはホームディレクトリの `~/.claude/` に設定ファイルを配置します。

```json
{
  "mcpServers": {
    "github": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-github"],
      "env": {
        "GITHUB_TOKEN": "your-token-here"
      }
    },
    "postgres": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-postgres"],
      "env": {
        "DATABASE_URL": "postgresql://localhost:5432/mydb"
      }
    }
  }
}
```

### 連携可能なサービスの例

- **GitHub Issues：** Issue内容を取得し、そのまま実装に着手
- **データベース：** スキーマ情報を参照しながらクエリやマイグレーションを作成
- **Sentry：** エラーログを取得して原因調査と修正を実施
- **Slack：** 進捗報告の自動送信

### 実践例：Issueから実装まで一気通貫

MCP経由でGitHub Issuesに接続している場合、以下のように依頼できます。

```
Issue #128 の内容を確認して、実装してください。
実装が完了したらテストも追加して、PRを作成してください。
```

Claude Codeが Issue の内容を取得し、要件を把握した上でコードを書き、テストを追加し、PRまで作成します。もちろん各ステップでユーザーの承認が入るため、意図しない操作は防げます。

さらに、GitHub Actionsに組み込めばCI/CDパイプラインでの自動化も可能です。PRが作成されたら自動でコードレビューを実行する、といったワークフローが構築できます。

## 権限モデル：安全に使うために知っておくべきこと

Claude Codeはファイル編集やシェルコマンド実行といった強力な操作を行います。そのため、操作の種類に応じた権限管理の仕組みが組み込まれています。

### 承認が求められる操作

デフォルトでは、以下の操作の実行前にユーザーの承認を求められます。

- ファイルの作成・編集・削除
- シェルコマンドの実行
- Git操作（commit、pushなど）

ファイルの読み取りや検索は承認なしで実行されます。

### 許可モードの設定

操作ごとの承認が煩わしい場合は、信頼度に応じて許可レベルを調整できます。

```bash
# 起動時に許可モードを指定
claude --allowedTools "Edit,Write,Bash(git *)"
```

この例では、ファイル編集・書き込みとgit系コマンドを自動承認し、それ以外のシェルコマンドは引き続き確認を求めます。

**注意：** 広範な権限を付与すると、意図しないファイル削除やコマンド実行のリスクがあります。最初はデフォルトの承認モードで運用し、信頼できる操作だけを段階的に許可していくのが安全です。

## 他ツールとの比較

主要なAIコーディングツールとの違いを整理します。

| 観点 | Claude Code | GitHub Copilot | Cursor | Aider |
|------|------------|----------------|--------|-------|
| 動作環境 | ターミナル | エディタ内 | 専用エディタ | ターミナル |
| 主な機能 | エージェント型操作 | コード補完+チャット | コード補完+チャット+エージェント | チャット+編集 |
| MCP連携 | あり | 限定的 | あり | 外部ツール連携あり（独自方式） |
| 公式最適化 | Anthropicモデルに最適化 | OpenAIモデル中心 | 複数モデル対応 | 複数モデル対応 |

Claude Codeの最大の強みは「エージェント性」です。指示を出せば自分でファイルを探し、読み、編集し、コマンドを実行します。補完ツールとは根本的にアプローチが異なります。一方、Cursorもエージェント機能やMCP連携を備えるなど各ツールは急速に進化しているため、最新の対応状況は各ツールの公式ドキュメントで確認することをおすすめします。

## 注意点とベストプラクティス

便利なツールですが、以下の点に注意が必要です。

- **コスト管理：** API従量課金の場合、長時間セッションではコストが膨らみます。Max Planの利用も検討してください
- **コード送信：** プライベートコードがAPIに送信されます。機密性の高いプロジェクトでは社内ポリシーを確認してください
- **変更の確認：** AIが自律的にファイルを編集するため、差分は必ず確認しましょう。デフォルトでは操作前に承認を求められます
- **セッション管理：** 長いセッションではコンテキストが膨らみます。`/clear` や `/compact` で定期的に整理しましょう
- **CLAUDE.mdの継続改善：** 出力品質が低いと感じたら、まず `CLAUDE.md` の記述を見直してください。特に「禁止事項」の追記が即効性があります

## まとめ

本記事で紹介した実践テクニックの要点を3つにまとめます。

- **CLAUDE.mdがすべての土台。** プロジェクト固有の情報、特に「やってはいけないこと」を充実させることで、AIの出力品質が飛躍的に向上する
- **エージェント型だからこそ、複数ファイルにまたがる作業で真価を発揮する。** リファクタリングは対象を分割して依頼し、テスト生成はモックの妥当性を重点確認しよう
- **MCP連携とCI/CD統合で、開発ワークフロー全体を効率化できる。** Issueの取得から実装、PR作成まで一気通貫の自動化が可能

まずは `npm install -g @anthropic-ai/claude-code` でインストールし、普段のプロジェクトで `claude /init` を実行してみてください。ターミナルでのAIペアプログラミング体験が、開発の進め方を変えてくれるはずです。
