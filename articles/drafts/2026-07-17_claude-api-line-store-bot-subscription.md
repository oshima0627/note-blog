# Claude API×LINEで店舗ボット副業：初期5万〜＋月額でMRRを積む設計図

「実装はできる。でも、いくらで売って、どう案件にすればいいか分からない」——このロードマップは、その一点を埋めるために書きました。ローカル店舗のLINE問い合わせ対応をClaude APIで自動化し、初期構築5万〜15万円＋月額1.5万〜3万円/店を、月額サブスク＝MRRとして積む二段構えの設計図です。金額はいずれも後述する市場相場からの推定レンジで、保証ではありません。個人が店舗にこの固定価格で成約した公開一次事例は確認できませんでした。それでも相場の構造は追えます。技術より、値付け・営業・契約の実務に踏み込みます。順に設計していきます。

## この副業で狙える収益の全体像とロードマップ

まず全体像を掴みます。この副業の収益は二段構えです。1つ目が納品時に一括で受け取る初期構築フィー、2つ目が毎月積み上がる月額運用サブスクです。前者はフロー収益、後者はストック収益（MRR）です。

金額の目安を置きます。初期5万〜15万円、月額1.5万〜3万円/店です。あくまで相場からの推定であり、保証ではありません。仮に月額2万円で10店契約できたとすると、月20万円のストックが立ちます。ここに初期フィーが都度乗ります。

ロードマップは5段階です。①業種を1つに絞ってテンプレ化する、②デモ店舗で動くボットを完成させる、③料金を固定パッケージにする、④最初の1件をモニター価格で取り事例化する、⑤同業種へ横展開し紹介でMRRを積む。この順に進めます。

重要なのは、最初から横展開を前提に設計する点です。1店ごとにゼロから作ると赤字です。テンプレ化で2店舗目以降の工数を圧縮し、初めてストックが利益に変わります。まずは1業種、1デモ、1モニターに集中します。

## なぜ今『ローカル店舗×LINE×Claude API』が個人に狙い目なのか

狙い目である理由は、競合の空白にあります。個人が業種特化テンプレで、生成AIによる自然な日本語応答を提供する層は、まだ薄いです。

市場の両端を見ます。ベンダーのAI型チャットボットは高機能ですが高額です。一方、汎用のSaaS型チャットボットは安価でも、ローカル店舗の細かな要望に弱いです。「常連さんの言い回しに合わせたい」「曖昧な問い合わせも汲み取ってほしい」といった声に、テンプレ回答では応えきれません。この中間が空いています。

ここでClaudeの日本語品質が差別化になります。敬語の自然さ、曖昧な問い合わせの汲み取りは、店舗接客と相性が良い領域です。整体の「腰が最近つらくて…」のような曖昧な入力を、適切な予約導線へ繋げられます。

もう1つの追い風が、非エンジニア店主の「丸投げ運用」ニーズです。多くの店主は設定も更新も自分ではやりたくありません。作って終わりではなく、運用まで代行するから月額が成立します。技術力よりも、この運用代行の姿勢が価値になります。個人でも入り込める理由がここにあります。

では、この市場を「単発の受託」で終わらせず「毎月積む収入」に変えます。その鍵が、次の二段構えの収益モデルです。

## 収益モデルは二段構え：初期構築フィーと月額運用サブスク

収益の設計図を具体化します。二段構えの中身を分解します。

初期構築フィーに含む作業は次の通りです。要件ヒアリング、LINE公式アカウントの開設支援、FAQ・シナリオ設計、Claude API連携、テスト、納品です。ここまでを一括で請求します。店ごとに1回だけ発生するフロー収益です。

月額運用サブスクに含む作業は次の通りです。稼働監視、FAQの更新、API従量課金の代行請求、軽微な改修、月次レポートです。これを毎月定額で請求します。積み上がるストック収益です。

この二段構えが効くのは、店舗側の心理に沿うからです。初期費用だけだと「作った後、誰が面倒を見るのか」という不安が残ります。月額運用をセットにすると、その不安が解けます。売り手にとっても、月額が積み上がるほど収入が安定します。

そして横展開の鍵が、業種別テンプレです。1業種でシナリオとプロンプトを固めておけば、2店舗目以降の初期工数を大きく圧縮できます。工数が下がるほど、初期フィーの利益率が上がり、月額はほぼ丸ごと利益に近づきます。テンプレ資産がMRRを支える構造です。

## いくらで売るか：固定パッケージの料金設計

値付けを決めます。都度見積もりは商談を長引かせます。固定パッケージにして即決を促します。

料金の目安は次の通りです。いずれも相場からの推定であり、保証ではありません。

- 初期構築フィー：5万〜15万円/店
- 月額運用サブスク：1.5万〜3万円/店

この帯に根拠を置きます。ランサーズのLINE Bot開発案件は5万〜30万円帯が見られます。AI搭載LINE BotのBasicプランは5万円〜という水準があります。チャットボット全体では、シナリオ型が初期無料〜10万円・月数千円〜、AI型が初期20〜50万円・月数万〜10万円という相場です。BOXILは主要26サービスの費用相場を月額約24,000円と集計しています。これらは2026年7月時点の情報で、最新は各媒体で要確認です。

固定パッケージの組み方を提案します。例えば、初期8万円＋月額2万円を「標準プラン」に据えます。上に、多言語対応や予約システム連携を足した「上位プラン」を置きます。下に、FAQ応答だけの「お試しプラン」を置きます。3段階にすると、店主は「真ん中」を選びやすくなります。

注意点があります。月額にはAPI従量費が乗ります。この実費を月額に含めるのか、別途実費請求にするのかを、契約時に明記します。曖昧なまま始めると、利益が削られます。次章で実費を正確に押さえます。

## コスト構造を正確に押さえる：LINEとClaude APIの実費

ここは「利益が残る根拠」を確認するパートです。数字だけ押さえれば、実装の詳細は後から戻ってきて構いません。利益を守るには、実費の把握が先です。原価が読めない見積もりは危険です。

まずLINE公式アカウントの料金です。以下は税別、2026年7月17日時点で、最新は公式で要確認です。

- コミュニケーションプラン：月0円、無料メッセージ200通
- ライトプラン：月5,000円、5,000通
- スタンダードプラン：月15,000円、30,000通、超過3円/通〜

Messaging APIの利用自体は無料です。店舗のFAQ応答用途なら、コミュニケーション〜ライトで足りることが多いです。なお、2026年10月1日に追加メッセージ料金の改定が公式告知されています。ここは後述のリスク章でも触れます。

次にClaude APIの料金です。USD/100万トークン、input/outputの順、2026年7月17日時点で、最新は公式で要確認です。

- Opus 4.8：$5 / $25
- Sonnet 5：導入価格 $2 / $10（2026年8月31日まで、9月1日以降は $3 / $15）
- Haiku 4.5：$1 / $5

コストを最小化する鍵が2つあります。プロンプトキャッシュの読取は基本input単価の約0.1倍です。Batch APIは入出力50%割引です。店舗FAQボットは、Haiku 4.5＋プロンプトキャッシュで従量費を抑えられます。1店の月間APIコストは、数百円〜数千円規模を想定します。ただしトラフィック次第であり、必ず実測してください。

## 技術構成：Messaging API webhook＋サーバーレス＋Claude API＋プロンプトキャッシュ

技術の全体像を示します。難しくはありませんが、要点を外すと運用で詰まります。

処理の流れはこうです。LINEユーザーの発話が、Messaging APIのwebhookを通じてサーバーレス関数に届きます。関数がClaude APIを呼び、応答を生成し、LINEへ返信します。サーバーレスの選択肢は、Cloud Run、Lambda、Vercelなどです。無料枠〜月数百円で始められます。

webhook受信部は概念的に10行程度です。詳細は割愛しますが、骨子だけ示します。

```python
@app.post("/webhook")
def webhook(req):
    for event in req["events"]:
        if event["type"] == "message":
            user_text = event["message"]["text"]
            reply = call_claude(user_text)   # Claude API呼び出し
            line_reply(event["replyToken"], reply)
    return "OK"
```

品質と原価の両立には、プロンプト設計が効きます。店舗のFAQとシナリオを、システムプロンプト＋FAQに落とし込みます。これをプロンプトキャッシュに載せ、従量費を圧縮します。基準モデルはHaiku 4.5です。難しい問い合わせのときだけ、上位モデルへ切り替えます。

1つ、絶対に守る設計原則があります。予約確定など確実性が要る処理は、AIに任せない。ここはAIに丸投げせず、人手や既存の予約システムへ橋渡しします。AIは「案内と一次対応」に徹し、確定処理は人・システムが担う。この線引きがトラブルを防ぎます。

## まず売り物を作る：業種を1つに絞ったデモ店舗ボット

営業の前に、売り物を用意します。口頭説明だけでは店主は動きません。動くデモが最強の営業資料です。

最初にやるのは、業種を1つに絞ることです。飲食店、美容室、整体のいずれか1つに決めます。あれもこれもは禁物です。1業種に絞ると、FAQもシナリオも深く作り込めます。その深さがテンプレ資産になり、横展開の元手になります。

次に、架空のデモ店舗を1つ設定します。営業時間、メニュー、料金、予約方法、よくある質問を用意します。これをシステムプロンプトとFAQに落とし込みます。実在店の実績を騙ってはいけません。あくまでデモとして提示します。

FAQに何を入れるかを具体化します。飲食店なら、営業時間、定休日、席数、予約可否、コース内容、アレルギー対応、支払い方法が定番です。美容室なら、メニューと料金、指名の可否、所要時間、当日予約、キャンセル規定が要ります。整体なら、症状の相談受付、初回の流れ、料金、施術時間、駐車場の有無が挙がります。この定番項目を業種テンプレの骨格にします。

シナリオも設計します。問い合わせは「質問」と「予約意図」に大別できます。質問には即答し、予約意図はフォームや電話へ橋渡しします。曖昧な入力、例えば「今日空いてる？」には、確定を避けつつ予約導線を返します。この分岐をシステムプロンプトに明文化します。

デモの完成基準を決めます。よくある問い合わせ10〜20件に、自然な敬語で正しく答えられること。予約希望には、確定はせず「予約フォーム/電話へ橋渡し」する導線を返せること。営業時間外の対応が破綻しないこと。この3点が通れば、営業に出せます。

工数の目安は、テンプレ＋デモの完成に実働20〜40時間です。初期コストはLINE無料プランで開始でき、サーバーレスは無料枠〜月数百円、APIは少額です。初月は数千円で収まります。金銭リスクは小さく、投じるのは主に時間です。

デモが1つあれば、営業は「作れます」ではなく「これが動いています」に変わります。ここからが本題、最初の1件の取り方に入ります。

## 案件化の5ステップ：モニター1件から横展開・MRRまで

ここで案件化の全ステップを通しで並べます。迷ったらこの順に戻ります。

1. 業種を1つに絞り、テンプレ化する。FAQとシナリオの型を作る。
2. デモ店舗で動くボットを完成させる。完成基準を満たす。
3. 料金を固定パッケージにする。初期＋月額の3段階を用意する。
4. 最初の1件をモニター価格で取り、事例化する。実運用の声を得る。
5. 同業種へ横展開し、紹介でMRRを積む。テンプレで工数を圧縮する。

このステップの肝は、④で「事例」を作ることです。前述の通り、個人が固定価格で店舗に売った公開一次事例は確認できませんでした。だからこそ、あなた自身が最初の事例を作る価値があります。1件の実績が、次の商談の説得力を跳ね上げます。

⑤では、同じ業種内で横に広げます。整体で作ったなら、次も整体を狙います。業種を跨がないほど、テンプレが効き、工数が下がります。工数が下がるほど、月額はほぼ利益として積み上がります。ここでMRRが本格的に立ち上がります。

必要スキルを確認します。LINE Messaging API、サーバーレス、Claude API連携、プロンプト設計、そして非エンジニア向けのヒアリング、見積・契約・請求です。後半の営業・事務スキルが、実は成否を分けます。

## 最初の1件をどう取るか：モニター価格と営業チャネル

最初の1件が最難関です。実績ゼロで定価は通りにくいです。そこでモニター価格を使います。

モニター価格の考え方はこうです。初期フィーを通常より下げ、または初期無料に近づけ、月額は正規に近い額で受けます。狙いは利益ではなく、事例と運用ノウハウの獲得です。条件として「導入事例として匿名で紹介させてほしい」を必ず付けます。

営業チャネルを固有名詞で挙げます。オンラインとオフラインを併用します。

- 地域の個人店への直接営業（飛び込み・チラシ・メール）
- 紹介・口コミ（既存客からの横紹介）
- ランサーズ、クラウドワークス（案件応募）
- ココナラ（パッケージ出品）
- X（実装過程やデモの発信）
- note（本記事のような解説での集客）
- 商工会・交流会（地域の店主との接点）

受注が見えたら、要件定義の入り口を丁寧に踏みます。最初のヒアリングで確認するのは5点です。①よく来る問い合わせは何か、②現状は誰がどう対応しているか、③予約は電話かフォームか既存システムか、④更新したい情報の頻度、⑤NGな回答や触れてほしくない話題です。この5点が、そのままFAQとシナリオの元になります。ヒアリングは30分あれば足ります。

非エンジニアの店主には、専門用語を避けて説明します。「webhook」ではなく「LINEに来た質問を受け取る仕組み」と言い換えます。相手が理解できる言葉に翻訳する力が、受注率を左右します。

最初の1件は、直接営業か紹介が取りやすいです。デモを見せ、「あなたのお店用に、まず低価格で試しませんか」と提案します。以下は提案文のテンプレ例です。

> ○○店 ご担当者さま。LINEでお客さまの問い合わせに24時間自動で一次対応する仕組みをご提案します。営業時間・メニュー・よくある質問に、自然な言葉で自動返信し、予約はスタッフへお繋ぎします。今回はモニターとして初期費用を抑えてご導入いただけます。まずは実際に動くデモをご覧ください。

押し売りはしません。デモを見せ、判断を委ねます。

## 横展開でMRRを積む：2店舗目以降の工数圧縮

1件取れたら、増やし方が主戦場になります。ここで利益が決まります。

2店舗目以降の要は、初期工数の圧縮です。1店目で作ったテンプレを使い回します。システムプロンプトの雛形、FAQの構造、webhookとサーバーレスの構成、監視の仕組みを共通化します。店ごとに変えるのは、店名・営業時間・メニュー・FAQの中身だけにします。

工数が下がると何が起きるか。仮に1店目が実働30時間で、テンプレ流用で2店目が10時間に下がったとします。同じ初期フィーなら、時間あたりの利益は3倍に近づきます。これが横展開の経済です。

MRRの積み上げも見えてきます。仮に月額2万円で契約を1店ずつ増やし、5店で月10万円、10店で月20万円のストックになります。これは目安の計算例であり、成約は保証されません。それでも、フロー案件と違い、積んだ月額は翌月も残ります。

横展開を加速するのが紹介です。同業種の店主同士は繋がっています。1店で満足されれば、近隣の同業へ紹介が広がります。だから初期の運用品質に手を抜かないことが、結局いちばんの営業になります。テンプレで効率化しつつ、対応の丁寧さは落とさない。この両立が積み上げの条件です。

## 契約・免責・規約リスクを先に潰す

ここを先に潰しておけば、後発の同業者より安心して受注できる差別化になります。トラブルは、起きてから対処すると高くつきます。契約で先に潰します。

まずLINEの規約と審査を守ります。禁止業種は扱いません。スパム的な一斉送信はしません。なりすましもしません。店舗名義のアカウントを、あなたが代行運用する立場を、契約で明確にします。「誰のアカウントを、誰が、どこまで操作するか」を文書化します。

次にAIの精度です。生成AIの精度は100%ではありません。だから予約確定など確実性が要る処理は、人手や既存の予約システムに任せます。ボットが誤った案内をする可能性があることを前提に、免責条項を契約に明記します。「一次対応の自動化であり、最終確認は店舗が行う」と定めます。

料金・仕様の変化もリスクです。LINEは2026年10月1日に追加メッセージ料金の改定が告知されています。Claude APIはUSD建てで、値上げやモデル廃止が起きます。実際、Opus 4.1は2026年8月5日にAPI引退、Sonnet 5は9月1日に値上げ予定です。だからAPI従量費は実費連動で請求し、為替と価格改定を契約に織り込みます。

最後に個人情報です。顧客データは最小化します。ボットに不要な個人情報を持たせません。ログの保管方針も店舗と合意します。これらは2026年7月時点の情報で、最新は各公式で要確認です。

## よくある失敗と回避策

先人がいない領域だからこそ、想定できる失敗を先に潰します。

1つ目は、業種を絞らない失敗です。複数業種を同時に狙うと、テンプレが浅くなり、工数も下がりません。回避策は、最初は1業種に集中することです。横展開も同業種から始めます。

2つ目は、予約確定までAIに任せる失敗です。誤確定はクレームに直結します。回避策は、確定処理を人手や既存システムに橋渡しし、契約に免責を明記することです。

3つ目は、API従量費を月額に固定で丸め込む失敗です。トラフィックが増えると赤字化します。回避策は、実費連動での請求か、上限を設けた定額かを、契約で明確にすることです。まず実測してから決めます。

4つ目は、値崩れです。競合が増えれば価格は下がります。回避策は、汎用機能で戦わないことです。業種特化のテンプレと、運用の丁寧さで守ります。安さではなく「任せられる安心」で選ばれる状態を作ります。

5つ目は、作って終わりにする失敗です。納品後に放置すると、FAQが古くなり解約されます。回避策は、月額運用に月次レポートと更新を必ず含め、価値を可視化することです。ストックは、運用の継続でしか維持できません。
