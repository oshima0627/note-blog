# Claude Agent SDK構築代行で受託単価を上げる7ステップ｜1案件30万〜100万円の目安

「Claude CodeやAgent SDKで一通り作れる。なのに、いくらで請ければいいのか分からない」——もしそうなら、この記事はあなたのためのものです。

企業のFAQ一次対応・社内ナレッジ検索(RAG)・定型作業をClaude Agent SDKで自律エージェント化し、要件定義から納品まで一括で請け負う「構築代行」。狙う収益の目安は1案件30万〜100万円に保守月額を上乗せする二段構えです。時給・工数売りから抜け出すための値付け・見積・契約・受注チャネルを、7ステップで一気通貫に解説します。

先に正直に線を引きます。本記事の金額はすべて相場からの推定・目安で、個人が固定価格で30万〜100万円を成約した公開一次事例は確認できていません。だからこの記事は、相場を鵜呑みにするのではなく、値付けの根拠を自分で持つことをゴールにします。読み終える頃には、何を・いくらで・どう提案し・どう契約するかが一通りつながっているはずです。

## この記事の対象読者と、狙える収益の目安（1案件30万〜100万円）

この記事は、次のような人に向けています。Claude CodeやAgent SDKで実装はできるが、案件化と値付けが分からない人です。あるいは時給・工数売りから単価を上げたい受託者・副業エンジニアです。RAGやサポート自動化を「作れる」段階から「売れる」段階へ進めたい人が対象です。

狙える収益の目安は、1案件30万〜100万円です。内訳の考え方はこうです。小規模な業務フロー単位なら30万〜50万円、中規模なら50万〜100万円を軸に設計します。ここに運用・保守の月額を上乗せし、ストック収益を積む二段構えにします。

この金額は、小〜中規模の業務フロー単位のエージェント構築の相場からの推定レンジです。断定できる固定価格ではありません。公開されている一次データの多くは月額単価や企業向けの本格開発費です。個人が固定価格で成約した公開一次事例は表に出にくく、確認できていません。だからこそ、後述する見積の分解と契約設計で「値付けの根拠」を自分で持つことが重要になります。

読み終える頃には、主力メニューの決め方、工程別の見積、API代の料金設計、契約リスクの防ぎ方、受注チャネルまでが一通りつながります。まずは全体像をつかんでください。では、なぜ今この構築代行が高単価で成立するのか。理由は3つあります。ここが腑に落ちると、後の値付けにも自信を持てます。

## なぜ今『Claude Agent SDKでの構築代行』が高単価で受注できるのか

理由の1つ目は、道具が整ったことです。Claude Agent SDKは、2025年9月29日にClaude Code SDKから改称した正式なSDKです。PythonとTypeScriptで提供されます（出典：Anthropic公式 claude.com/blog、code.claude.com/docs、2026年7月時点確認）。

このSDKは、エージェント構築に必要な要素を標準搭載しています。ファイルシステムへのアクセス、bashコマンドの実行、MCP連携、サブエージェント、セッション・コンテキストの圧縮です。つまり、自律的に動くエージェントの土台を自前でゼロから組む必要が減りました。実装者にとっては、価値ある部分に時間を集中できるということです。

理由の2つ目は、需要と供給のギャップです。企業側にはFAQ対応・社内ナレッジ検索・定型作業の自動化ニーズが強くあります。一方で、それを実装できる内製人材は不足しがちです。ここに外部委託の余地が生まれます。

理由の3つ目は、情報の空白です。Agent SDKを使った実装の解説は増えています。しかし「値付け・契約・API従量課金の顧客転嫁」まで一気通貫で扱う情報は多くありません。この3点を押さえた提案ができる人は、まだ少数です。先行して型を持つほど有利になります。

なお、SDKの仕様や機能は変わりやすい領域です。提案や実装の前には、最新情報を公式（code.claude.com）で必ず確認してください。本記事の記述は2026年7月時点の理解に基づきます。

## 収益ゴール：時給売りをやめ『成果物単位30万〜100万円＋保守月額』にする

収益モデルを明確にします。柱は受託開発です。要件定義・設計・実装・納品を一括で請け負います。これに運用・改修・保守の継続契約を組み合わせます。1案件完結型を軸に、保守月額でストック収益を上乗せする二段構えです。

ここで、工数売りと成果物売りの違いを押さえてください。工数売り（人月）は、働いた時間に単価を掛けて請求します。単価の天井は市場相場に縛られます。成果物売りは、納品する価値に対して価格を付けます。効率化して早く作れば、時給換算はむしろ上がります。単価の設計権を自分が持てるのが成果物売りです。

相場の参考値を挙げます。ランサーズのプロフェッショナルエージェントには、生成AIを活用したRAG構築支援で報酬70万〜90万円級の案件掲載があります（lancers.jp、2026年7月10日確認、月額と推定）。受託会社のブログには、業務フロー単位の自動化が50万円〜、PoCが150万〜500万円、本格実装が1500万円〜という記載も見られます（二次情報、2026年7月10日確認）。

これらは二次情報や相場からの推定を含みます。そのまま自分の固定価格として断定はできません。ただ、小〜中規模案件で30万〜100万円というレンジは、現実味のある目安として置けます。次章から、この金額を「どう分解し・どう根拠づけるか」を具体化します。テンプレートは7ステップの中で順に示します。金額の確定は誰にも渡せません。渡せるのは、自分で根拠を組み立てる型です。ここから7ステップで、その型を1つずつ作ります。まずはステップ1、扱う商品を1つに絞るところからです。

## ステップ1：提供メニューを1つに絞って『型』にする

最初にやるのは、提供メニューを1つに絞ることです。候補は3つです。社内ナレッジ検索(RAG)の構築、FAQ・問い合わせ一次対応の自動化、定型作業の自律エージェント化です。この中から主力を1つ選びます。

なぜ絞るのかを説明します。メニューを絞ると、提案・見積・納品が定型化します。毎回ゼロから設計しなくてよくなります。結果として、作業時間が読め、単価と利益率が安定します。「何でもやります」は、値付けも品質も安定しません。まず1つで型を作るのが近道です。

型化の中身は、Claude Agent SDKでの再利用テンプレート実装です。共通で使う部分を部品化しておきます。例えばRAGを主力にするなら、次のような要素を型に組み込みます。MCP連携でデータソース（社内ドキュメントやDB）につなぐ部分です。サブエージェントで「検索担当」と「回答生成担当」を分ける構成です。セッション・コンテキスト圧縮で長い会話でも破綻しにくくする部分です。これらはSDKの標準機能なので、型に落とし込みやすいです。

選び方の目安を挙げます。自分が実装経験を持ち、デモを短時間で作れるものを選んでください。加えて、企業の困りごとが分かりやすいものが有利です。FAQ自動化や社内検索は、成果を説明しやすいテーマです。まずは1つに決めて、次のステップでデモ資産に落とし込みます。

## ステップ2：受注前に効く『デモ資産（ポートフォリオエージェント）』を作る

実績ゼロの段階で信頼を作る鍵が、デモ資産です。デモ資産とは、完動状態のポートフォリオエージェントです。公開情報や自分の資料をナレッジ源にしたRAGエージェントを、実際に動く形で用意します。

見せ方も設計します。動く画面の動画やGIFを撮り、構成図をセットにします。提案の場で「動くもの」を見せられると、説明の説得力が変わります。口頭の説明よりも、質問に答えるエージェントを実演するほうが伝わります。

ここで差別化の軸になるのが、ハルシネーション対策です。デモの段階から、出典表示と回答根拠の提示を組み込みます。「この回答はどの資料に基づくか」を示せるエージェントは、企業の不安に直接応えます。AIの誤答リスクを気にする発注者ほど、この設計を評価します。最初から品質の証拠を見せる、という考え方です。

コストの目安を示します。初期コストはほぼゼロ〜数千円/月です。SDK自体は無料です。開発検証のAPI利用料は少額に収まります（後述のとおり、サポート1万件処理でHaiku 4.5が約$37という公式試算があり、検証段階はこれよりさらに小さくなります）。時間については、メニュー型化とデモ資産作成で実働20〜40時間程度が目安です。ただしこの工数はスキル前提により変動する推計です。自分の習熟度に合わせて見積もってください。デモ資産で信頼は作れます。次はいよいよ値付けです。30万〜100万円をそのまま提示すると、必ず値切られます。ここからは、そのまま提案書に貼れる見積・契約・提案文のテンプレートを順に渡していきます。

## ステップ3：30〜100万円を『工程別に分解』した見積もりテンプレートを作る

30万〜100万円という金額を、そのまま提示してはいけません。根拠が見えないと値切られます。そこで、見積を工程別に分解します。分解する工程は次のとおりです。

- 要件ヒアリング：業務と課題の把握、対象範囲の確定
- 設計：エージェント構成、データ連携方式、精度目標の定義
- 実装：Agent SDKでの構築、RAGの検索・回答生成
- データ連携・チューニング：ナレッジ取り込み、精度調整
- テスト：想定質問での検証、誤答パターンの確認
- 納品・ドキュメント：操作手順、構成図、引き継ぎ資料

プランは2つ用意します。小規模プランは30万〜50万円です。中規模プランは50万〜100万円です。金額配分の考え方はこうです。小規模では、実装とチューニングに比重を置きます。中規模では、要件ヒアリングと設計、テストの比重が上がります。対象範囲が広いほど、上流と検証の工数が増えるためです。

重要な原則が1つあります。API従量課金は本体価格に含めないことです。運用で継続的に発生するAPI代を一括の本体価格に丸呑みすると、赤字リスクが生まれます。API代は「実費＋運用保守月額」として分離します。詳細は次のステップで扱います。

そのまま提案書に貼れる見積テンプレートを置きます。金額は案件規模で変わるため、必ず自分の見積に合わせて調整してください。

> 【お見積り（構築代行・小規模プラン例）】
> ・要件ヒアリング：◯円
> ・設計（エージェント構成・精度目標定義）：◯円
> ・実装（Claude Agent SDK構築・RAG検索/回答生成）：◯円
> ・データ連携・チューニング：◯円
> ・テスト・精度検証：◯円
> ・納品・ドキュメント：◯円
> 小計（本体・一括）：30万〜50万円（案件規模により変動）
>
> ※上記に運用API実費および保守月額は含みません（別掲）。
> ※金額は業務範囲・データ量・精度目標により調整します。相場からの推定・目安です。

見積の透明性は、そのまま信頼になります。工程を分けておくと、値引き交渉が「全体を値切る」から「工程単位で範囲を調整する」に変わります。これは受注側にとって有利な議論の土俵です。テンプレートは案件ごとに数字を埋めて使ってください。

ここから先の見積テンプレート・契約条項文例・提案文テンプレートは、そのままコピーして数字を埋めれば使える形にしています。「作れる」を「売れる」に変える実務パーツは、この見積の型から始まります。

## ステップ4：API代（運用コスト）を試算し、顧客に転嫁する料金設計にする

運用のAPI代は、契約で顧客負担を明確にします。本体価格に丸呑みすると、使われるほど赤字に近づきます。設計の型は「実費＋保守月額」として分離することです。

概算の目安を、2026年7月時点の公式料金で示します。以下の料金はいずれも100万トークンあたりです。最新は必ず公式（platform.claude.com）で確認してください。ここは顧客への請求で損しないための章です。数字を暗記する必要はなく、本体価格と分けて実費＋月額で請求するという結論だけ持ち帰れば十分です。

- Opus 4.8：入力 $5 / 出力 $25
- Sonnet 4.6：入力 $3 / 出力 $15
- Sonnet 5（導入価格）：入力 $2 / 出力 $10（2026年8月31日まで。以降は $3 / $15 に戻ります）
- Haiku 4.5：入力 $1 / 出力 $5

処理量の感覚をつかむ目安として、サポート1万件の処理はHaiku 4.5で約$37という公式試算があります。これは月額換算のイメージとして提案に使えます。実際の費用は、質問の長さやナレッジ量、モデル選択で変動します。

モデルの使い分けが、コスト設計の肝です。精度を重視する工程はSonnetやOpusを使います。コストを抑えたい大量処理はHaikuを使います。加えて、プロンプトキャッシュを活用します。キャッシュヒット時の入力は、通常の入力の0.1倍で計算されます。同じ前提文を繰り返し送る用途では、ここで大きく圧縮できます。

見積に織り込む注意点が1つあります。Opus 4.7以降は新しいトークナイザーになり、同一テキストでも約30%多くトークンを消費します。トークン数が増えれば費用も増えます。Opus系を使う設計では、この増加分を試算に含めてください。

料金は変わりやすい領域です。Sonnet 5の$2/$10は2026年8月31日までの導入価格です。提案書には必ず「2026年7月時点・最新は公式(platform.claude.com)要確認」と明記してください。基準日を書くだけで、後のトラブルを大きく減らせます。

## ステップ5：契約リスク（著作権・精度保証・PoC止まり）を防ぐ雛形を用意する

高単価案件ほど、契約で身を守る必要があります。防ぐべきリスクは3つです。著作権、精度保証、PoC止まりです。

まず著作権です。AI生成物そのものには、原則として著作権が発生しないとされています（文化庁見解ベース、2026年7月時点の理解）。だからこそ、契約で成果物の権利帰属を発注者に明記します。あわせて、利用したAIツール（モデル名・バージョン・利用範囲）を特記して開示します。そして、AI生成物は人間が最終確認する前提を明記します。

次に精度保証です。LLMは100%の正答を保証できません。だから「保証」という言葉で約束してはいけません。代わりに「目標精度＋改善サイクル」として定義します。目標値を示し、達成に向けて改善を回す枠組みにするのです。

3つ目はPoC止まりです。PoC（検証）で終わる案件は少なくありません。対策は、PoCと本開発を段階契約に分けることです。PoC単体でも収益化できる価格設計にしておけば、本開発に進まなくても赤字になりません。

そのまま使える契約条項の文例を置きます。これは雛形です。実際の契約は、必ず法務・契約の専門家に確認してください。

> 【契約条項の文例（抜粋・要専門家確認）】
> 第◯条（権利帰属）本業務の成果物に関する権利は、検収完了をもって発注者に帰属する。ただしAI生成物そのものに著作権が発生しない場合があることを両者は確認する。
> 第◯条（利用ツールの開示）受注者は、成果物の作成に用いたAIモデルの名称・バージョン・利用範囲を発注者に開示する。
> 第◯条（人間による確認）AIによる生成物は、納品前に人間が内容を確認・調整するものとし、受注者はその責任を負う。
> 第◯条（精度）本エージェントの応答精度は目標値を◯◯とし、受注者は改善サイクルの実施に努める。受注者は完全な正答を保証しない。
> 第◯条（段階契約）本契約はPoC（検証）フェーズと本開発フェーズに分け、各フェーズの成果と対価を別途定める。

この5点を雛形に入れておくだけで、後のトラブルの多くを未然に防げます。特に精度と著作権は、AI案件で揉めやすい論点です。最初に言葉で握っておくことが、信頼にもつながります。

## ステップ6：受注チャネルを『応募系』と『発信系』の2系統で回す

受注チャネルは、2系統で回します。応募系と発信系です。片方に偏らず両輪にすると、機会が安定します。

応募系は、案件が集まる場所に応募する動き方です。具体的には、ランサーズ、クラウドワークス、レバテック等のフリーランスエージェントです。RAG・生成AI・AIエージェントの案件に応募します。ランサーズはAI関連の案件が1,700件超あり、プロフェッショナルエージェント経由でRAG構築70万〜90万円級の掲載実績が見られます（2026年7月時点確認の目安）。クラウドソーシングは、10万円以上の案件だと手数料面で有利になりやすいという比較記事もあります。

発信系は、自分の存在を見つけてもらう動き方です。X（旧Twitter）、note、Zennでデモやノウハウを発信します。ステップ2で作ったデモ資産が、ここで効きます。動く成果物を見せて、直接相談や紹介につなげます。実は、最も高単価になりやすいのは、知人や既存取引先からの紹介です。発信は、その入り口を広げる投資でもあります。

応募時にそのまま添えられる提案文テンプレートを置きます。案件内容に合わせて具体化してください。

> 【提案文テンプレート（応募系）】
> はじめまして。Claude Agent SDKを用いた業務自動化・RAG構築を専門にしております。
> ・実装内容：御社の[対象業務]を対象に、社内ナレッジ検索/FAQ一次対応エージェントを構築します。
> ・差別化：回答に出典・根拠を表示し、人間の最終確認を前提とした設計で誤答リスクを抑えます。
> ・進め方：まずPoC（小規模検証）を[金額・期間]で実施し、効果を確認のうえ本開発へ進みます。
> ・費用：本体一括[金額]＋運用API実費＋保守月額[金額]（料金は2026年◯月時点、詳細は別途お見積り）。
> 動作デモ（動画）と構成図をご用意しています。ご希望であればすぐに共有します。

案件数や単価は変動します。上記の数値は2026年7月時点の目安として扱ってください。応募文には、必ずデモ資産へのリンクを添えると効果的です。

## ステップ7：小さく受注→事例化→単価を上げるループを回す

単価は、いきなり上限を狙わないほうが安全です。まず小さく受注し、確実に納品します。初回は小規模プラン（30万〜50万円）で、リスクを抑えて実績を作ります。無理に大型案件を取って納品でつまずくと、信頼を失います。

次に、納品した仕事を事例化します。顧客名を出せない場合でも、内容は蓄積できます。「どの業務を・どう自動化し・どんな成果が出たか」を記録します。匿名可の範囲で、業種・課題・打ち手・結果を整理しておきます。これが次の提案の説得材料になります。

実績が積み上がったら、単価を引き上げます。中規模プラン（50万〜100万円）に加え、保守月額を組み込みます。ここでストック収益が育ちます。1案件完結だけでなく、継続収入の基盤ができていきます。

1つだけ、絶対のルールがあります。実績の脚色や架空事例を作らないことです。「◯社で成約」「月◯万円達成」といった未確認の実績を書いてはいけません。信頼の面でも、プラットフォーム規約の面でもNGです。事実だけを、事実として積み上げてください。地味でも、これが長く稼ぎ続ける唯一の土台です。

## つまずきやすい注意点とハルシネーション対策

最後に、つまずきやすい点をまとめます。

1つ目は陳腐化リスクです。ノーコードのRAGツールやマネージドエージェントが普及すると、単純な構築は価格競争になりえます。対策は、価値の軸を上流と伴走に置くことです。業務理解・要件定義・運用改善は、ツールが自動化しにくい領域です。「作る」だけでなく「業務を良くする」に軸足を移してください。

2つ目は、料金・仕様の変わりやすさです。Claude APIの料金やSDKの仕様は更新されます。提案書と契約書には、必ず基準日と「最新は公式(platform.claude.com / code.claude.com)要確認」を明記してください。基準日のない数字は、トラブルの火種になります。

3つ目は、これまで述べた3大リスクの再確認です。PoC止まりは段階契約で防ぎます。精度クレームは「目標精度＋改善サイクル」で防ぎます。API赤字は「実費＋保守月額」の分離で防ぎます。この3つの設計は、契約前に必ず入れてください。

4つ目は、規約と事実の扱いです。クラウドソーシング各社の規約と、Anthropicの利用規約を守ってください。そして、数値・固有名詞・事例には出典と基準日を添えてください。確認できない情報は「目安」と書くか、正直に「確認できていない」と書きます。これはあなた自身の信頼を守る行為です。

## よくある質問（実装スキルはあるが受注が初めての人向け）

Q. 実績ゼロでも30万円で受注できますか。
A. 相場からの推定として、可能性はあります。鍵はデモ資産と小規模プランです。動く成果物を見せ、小さく確実に納品して信頼を作ります。いきなり高額を狙わず、実績を積む順番が現実的です。

Q. API代で赤字になりませんか。
A. 料金を本体に丸呑みしなければ、リスクは抑えられます。「実費＋保守月額」で分離します。加えて、モデルの使い分けとプロンプトキャッシュ（キャッシュヒットは入力の0.1倍）で費用を圧縮します。

Q. 精度のクレームが怖いです。
A. 契約で「保証」を約束しないことです。LLMは100%正答を保証できません。「目標精度＋改善サイクル」として定義し、事前に握っておきます。これで期待値のズレを防げます。

Q. 副業でも回せますか。
A. メニューの型化と段階契約で工数を抑えれば、可能性はあります。ただし必要工数は前提スキルにより変動する推計です。まずはデモ資産作りに実働20〜40時間を見込み、無理のない範囲で始めてください。
